Abstract

Auto Stitch is a full-stack, AI-powered, multi-boutique fashion marketplace specifically designed for the Pakistani market. It delivers virtual try-on, part-level outfit customization, and intelligent shopping assistance within a single privacy-aware platform. Users can explore products from multiple boutiques, request custom modifications targeting specific garment regions such as neckline, sleeves, or hemline using reference images, receive competitive price bids from boutiques, and preview outfits on their own body using an AI-based virtual try-on pipeline. The pipeline follows a four-stage asynchronous process using pre-trained models: pose estimation using MediaPipe and HRNet, human parsing using SCHP, garment warping using PF-AFN, and generative refinement using LaDI-VTON. The platform offers personalized recommendations powered by FashionCLIP (built on CLIP) embeddings stored in PostgreSQL with pgvector, a RAG-enabled chatbot using LangChain with GPT-3.5-turbo, secure payment integration via Stripe (sandbox/test mode), and a comprehensive admin dashboard for managing listings, orders, bids, and quality control. User privacy is enforced through explicit consent-based image handling, ephemeral S3 processing, and encrypted storage using AWS SSE-KMS, with privacy principles aligned to Pakistan's Personal Data Protection Bill (PDPB). The system is built on a modular microservice architecture connecting a React frontend, Node.js/Express API server, FastAPI ML microservices, Celery + Redis async task queue, PostgreSQL and MongoDB databases, and AWS S3 object storage, all containerized with Docker and deployed via GitHub Actions CI/CD pipelines.

1. Introduction
Auto Stitch is an AI-powered, multi-boutique fashion marketplace designed for Pakistan's growing e-commerce market, where existing platforms offer only basic catalogue listings without AI augmentation leading to high product return rates (15-40%), low customer confidence, and limited boutique visibility. The platform addresses this gap by integrating a four-stage virtual try-on pipeline (MediaPipe/HRNet, SCHP, PF-AFN, LaDI-VTON), part-level garment customization via ControlNet and InstructPix2Pix, FashionCLIP-based personalized recommendations, a RAG chatbot (LangChain + GPT-3.5-turbo), boutique bidding, Stripe sandbox payments, and privacy-first design -- all within a modular microservice architecture built on React, Node.js/Express, FastAPI + PyTorch, PostgreSQL + MongoDB, AWS S3, Celery + Redis, and Docker, serving Customer, Boutique Owner, and Administrator roles. This SRS document defines the complete software requirements, use cases, and design artifacts for the Auto Stitch platform as scoped for a three-person FYP team at FAST-NUCES.

Following are the objectives of the Auto Stitch platform:
Enable customers to virtually try on garments using a four-stage AI pipeline (MediaPipe/HRNet, SCHP, PF-AFN, LaDI-VTON) with pre-trained weights only.
Provide part-level garment customization (neckline, sleeves, hemline, embroidery) via ControlNet and InstructPix2Pix using customer-uploaded reference images.

Containerize all services with Docker and automate deployment via GitHub Actions CI/CD pipelines.

Payments are handled via Stripe checkout in sandbox/test mode, with no real money processed during the FYP phase.

Boutique Bidding System (customization requests with competitive bids)

3.2. Functional Requirements
3.2.1. User Account Management
The system shall allow users to register with email/password or via Google and Facebook OAuth 2.0, with input validation and JWT-based authentication (15-min access tokens, 7-day refresh tokens). RBAC enforces three roles: Customer, Boutique Owner, and Administrator. Users may update profiles, reset passwords via email, and optionally enable TOTP 2FA (mandatory for Administrators). Accounts are locked after 5 failed login attempts with exponential backoff, refresh tokens rotate on every use, and per-user consent flags in MongoDB govern image processing and retention behaviour.
3.2.2. Product Browsing & Search
The system shall display a paginated product catalogue with filtering by category, price, boutique, color, size, and material, and support full-text keyword search via Elasticsearch with automatic index sync on every product change. Product detail pages include multiple images, size charts, customization options, boutique information, and customer reviews. Customers can maintain a wishlist, view FashionCLIP-based related products, and perform cross-modal (text-to-image and image-to-image) search using pgvector similarity queries.


3.2.3. AI Virtual Try-On
The system shall allow authenticated customers to upload a full-body photo (JPEG/PNG, max 10 MB) after providing explicit consent, and enqueue a try-on job via Celery + Redis. The four-stage pipeline extracts body pose keypoints (MediaPipe/HRNet), segments body regions (SCHP), warps the garment to fit the body shape (PF-AFN), and synthesizes the final composite try-on image (LaDI-VTON) with face masking applied throughout. Results are stored in AWS S3 with SSE-KMS encryption and a 1-hour ephemeral TTL. The user is notified on completion, can cancel in-progress jobs, and post-try-on feedback (1-5 stars) is collected for future model improvement.
3.2.4. Garment Customization
The system shall allow customers to initiate a part-level customization request by selecting target regions (neckline, sleeves, hemline, embroidery) and uploading reference images (JPEG/PNG, max 5 MB each). ControlNet applies structural conditioning and InstructPix2Pix handles texture and style transfer asynchronously via Celery. Customers review the generated preview alongside the original and either submit it as a formal request to boutiques or discard it. Each revision stores full provenance metadata, and a Socket.IO messaging channel facilitates direct communication between customer and boutique.
3.2.5. Boutique Bidding System
The system shall allow customers to broadcast approved customization requests to all boutiques in the relevant category. Boutique owners submit bids with a quoted price, estimated timeline, and optional portfolio samples. Customers compare bids in a sorted view, accept one to create a confirmed order, or reject/let requests expire after 72 hours. Boutique reputation scores (0.0-5.0) are computed from ratings, completion rate, on-time delivery, and bid acceptance rate. Bid status updates are communicated in real time, and undelivered accepted orders trigger an admin-mediated dispute and refund workflow.
3.2.6. Product & Catalogue Management
The system shall allow boutique owners to create, update, and delete product listings with up to 8 images stored in AWS S3 via CloudFront CDN, and support bulk CSV import. FashionCLIP embeddings (512-dim) are auto-generated for new product images and stored in PostgreSQL with pgvector. New listings enter an admin moderation queue before becoming publicly visible. Boutique onboarding requires KYC documents (CNIC, business registration certificate), and shipping zone configuration supports integration hooks for TCS, Leopards Courier, and Pakistan Post rate APIs.
3.2.7. Order Management
The system shall support a full cart-to-delivery workflow with server-side validated totals, a coupon engine, and database advisory locks to prevent overselling. Order statuses progress through Placed, Accepted, In Production, Ready to Ship, Shipped, Delivered, Cancelled, Dispute, Refund Requested, and Refunded, with invalid transitions returning HTTP 400. Customers can cancel orders, view order history, and initiate return/refund requests with evidence images. Boutique owners update statuses with optional tracking information, and customers receive in-app and email notifications at each stage.
3.2.8. Payments (Stripe Sandbox)
The system shall integrate Stripe Checkout in sandbox/test mode, supporting test card payments, webhook event handling for payment confirmation, failure, and refunds, and a 3-installment checkout plan (33%/33%/34% at 0/30/60 days) scheduled via node-cron. All payment API calls include idempotency keys, webhook signatures are verified via HMAC, and transaction records are stored in PostgreSQL with Stripe payment intent IDs. No real money is processed during the FYP phase.
3.2.9. Recommendations Engine
The system shall generate personalized product recommendations using 512-dim FashionCLIP embeddings stored in PostgreSQL with pgvector HNSW indexing for sub-100ms nearest-neighbor queries. User preference vectors are built from browsing history, wishlist items, and past purchases, with a popularity-based cold-start fallback for new users. Real-time session signals (views, wishlist additions, dwell time) are tracked in Redis for dynamic re-ranking, and embedding upserts are triggered by Celery on product updates with nightly consistency checks.
3.2.10. RAG Chatbot
The system shall provide a floating chatbot widget powered by LangChain and GPT-3.5-turbo, using product catalogue data as the RAG knowledge base. It supports multi-turn conversation context, clickable product links in responses, and rate limiting at 30 queries per user per hour. When retrieval confidence drops below 0.6, a fallback message is returned. The chatbot never directly executes sensitive actions; all confirmed intents are routed through signed backend API calls using the user's JWT session. Conversation history is limited to the 10 most recent turns and stored in MongoDB.
3.2.11. Admin Moderation & Dashboard
The system shall provide administrators with an exclusive dashboard for approving or rejecting product listings, managing user and boutique accounts, reviewing flagged content, and monitoring platform-wide metrics (total users, active boutiques, total orders, pending moderation items). The moderation queue operates under a 24-hour SLA with visual escalation indicators, supporting bulk approve/reject actions with templated rejection messages. All administrative actions are recorded in immutable audit logs with full before/after state, IP address, and user agent metadata.
3.2.12. Notifications
The system shall deliver in-app notifications via Socket.IO for order updates, bid responses, try-on completion, and customization results, and email notifications via SendGrid for account and payment events, with browser push support via the Web Push API. Notification events are published through Redis Pub/Sub with per-user preference filtering before dispatch, and failed deliveries are re-queued with exponential backoff up to 5 retries with a 30-minute maximum delay.
3.2.13. Analytics Dashboard
The system shall provide boutique owners with sales analytics (revenue, order count, average order value, top-selling products) and customer engagement metrics, and administrators with platform-wide analytics including user and boutique growth, transaction volume, AI feature usage, and boutique performance rankings. Data is visualized with charts on the frontend and exportable as CSV, with all export endpoints rate-limited and PII masked for boutique-facing exports. Materialized PostgreSQL views compute MAU/DAU, AOV, and conversion rates hourly, and Prometheus/Grafana monitor AI pipeline health metrics.
3.3. Non-Functional Requirements
3.3.1. Performance Requirements
The API server shall respond to standard CRUD requests within 200 ms under normal load (up to 100 concurrent users).
FashionCLIP recommendation queries using pgvector HNSW index shall return results within 100 ms.
AI virtual try-on pipeline (all four stages) shall complete within 60 seconds per request on a single GPU instance.
Garment customization (ControlNet + InstructPix2Pix) shall complete within 45 seconds per request.
The chatbot (GPT-3.5-turbo) shall return responses within 5 seconds per query.
Product catalogue pages shall load within 2 seconds on a 4G mobile connection.
The system shall support at least 100 concurrent users during the FYP demonstration phase.
3.3.2. Safety Requirements
The system shall require explicit user consent before processing any uploaded images through AI models.
The system shall apply face masking during ML inference to prevent facial data from being used in garment-related processing.
Processing artifacts and ephemeral images shall be automatically deleted from S3 after a 1-hour TTL.
The system shall validate all uploaded images for file type (JPEG/PNG only) and size limits (10 MB for try-on, 5 MB for customization references) to prevent malicious uploads.
The system shall implement rate limiting (Redis-backed) on API endpoints to prevent abuse and denial-of-service: 100 requests/min for standard endpoints and 10 requests/min for AI inference endpoints.
3.3.3. Security Requirements
All API communication shall use HTTPS/TLS encryption.
User passwords shall be hashed using bcrypt with a minimum cost factor of 10.
Authentication shall use JWT tokens with short-lived access tokens (15 min) and longer refresh tokens (7 days), with secure httpOnly cookie storage.
All user inputs shall be validated and sanitized using Zod schema validation on the backend to prevent injection attacks (OWASP A03).
AWS S3 objects shall be encrypted at rest using SSE-KMS (Server-Side Encryption with Key Management Service).
Role-based access control (RBAC) shall be enforced on all API endpoints; unauthorized access attempts shall return HTTP 403.
Nginx reverse proxy shall serve as the API gateway with rate limiting and request size restrictions.
Database connections shall use parameterized queries to prevent SQL injection.
CORS policy shall restrict API access to the registered frontend origin only.
Administrative actions shall be recorded in an immutable audit log in PostgreSQL.
The application shall implement CSRF protection (double-submit cookie pattern via csrf-csrf) on all state-changing endpoints.
The Node.js API server shall apply Helmet.js security headers on all responses.
All payment gateway webhook payloads shall be verified via HMAC signature validation before processing.
Idempotency keys shall be included on all payment and order creation API calls to prevent duplicate transactions.

3.3.4. Software Quality Attributes
3.3.4.1. Maintainability
The codebase shall follow a modular microservice architecture with clear service boundaries (React frontend, Node.js API, FastAPI ML services) to enable independent development and deployment.
3.3.4.2. Scalability
The system shall be containerized with Docker to allow horizontal scaling of individual services in future deployments (auto-scaling deferred to future work).
3.3.4.3. Testability
The system shall achieve a minimum of 70% unit test coverage on backend API routes and ML service endpoints, with integration tests for critical workflows (registration, try-on, order placement).
3.3.4.4. Reliability
The Celery + Redis task queue shall implement retry logic with exponential backoff for failed AI processing jobs (max 3 retries).

3.3.4.5. Usability
The React frontend shall follow responsive design principles, providing usable interfaces on screen widths from 320px (mobile) to 1920px (desktop).
3.3.4.6. Portability
All services shall be containerized with Docker and orchestrated via Docker Compose, ensuring consistent deployment across development, staging, and production environments.
3.3.4.7. Privacy
The system shall align with PDPB principles including purpose limitation, data minimization, storage limitation, and user consent, as documented in the Privacy Impact Assessment.
